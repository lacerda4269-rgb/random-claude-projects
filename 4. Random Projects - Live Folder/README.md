# 4. Random Projects - Live Folder

**Owner:** Sage (CEO)
**Purpose:** Isolated workspace for active project-specific working records — files that are live, being worked on, and don't belong in a standing agent's folder or the Master Library.

---

## What This Folder Is

The Live Folder is an isolated working area that runs alongside the main structure without polluting it. It holds active records that are specific to the current project — files being actively built or referenced that have no permanent home elsewhere in the structure. The rest of the package stays clean because in-progress work lives here.

**Blank version:** Empty at baseline beyond this README. The folder name (`Named Project - Live Folder`) is a placeholder — rename it to `[Project Name] - Live Folder` at Phase 0 setup. Phase 0 or the first active phase populates this folder as working files are created.

---

## What Goes Here

- Active project-specific working records with no other designated home
- In-progress build reports and indices being actively updated
- Any project-specific working file that must be isolated from the standing structure

## What Does NOT Go Here

- Cleared and cataloged resources (those go to the Master Library)
- Completed deliverables with a permanent destination (those go to their designated home)
- Files that should travel with this template to *future* projects (those belong in standing folders or the ML)

---

## Governance

| Rule | Detail |
|------|--------|
| **Naming convention** | Rename from `Named Project - Live Folder` to `[Project Name] - Live Folder` at Phase 0 setup |
| **Lifecycle** | Files here are active. When a file's work is complete, it moves to its permanent destination or is archived |
| **Template clean** | Nothing in this folder carries to the next project's baseline — this folder starts empty every time |

---

## Shared / Cross-Agent Handoff (on-demand)

**Owner:** Sage (CEO)
**Purpose:** Cross-agent collaboration space — artifacts that belong to the handoff *between* agents, not to any single agent's folder.

This is an **on-demand subfolder** — it is **not pre-created**. It is created on the first cross-agent handoff need, the same on-demand philosophy as the agent residency model (only Sophia ships live; every other agent and shared structure spins up when the project actually requires it). When two or more agents working on connected objectives need a shared layer, Sage creates the subfolder here.

**Classic use case:** Nick (NinjaTrader) builds a trading strategy. Pete (Python) builds the backtesting engine for it. The interface spec — what the strategy outputs, what the engine consumes — lives here, not in Nick's folder or Pete's folder.

### What Goes Here

- Interface specifications between agents
- Shared deliverables with no single agent owner
- Cross-agent handoff documents
- Anything that two or more agents must reference or maintain together

### What Does NOT Go Here

- Individual agent work products — those stay in the agent's folder
- Cleared and cataloged resources — those go to the Master Library
- Personal agent notes — those stay in the agent's workspace

### Governance

| Rule | Detail |
|------|--------|
| **Created by** | Sage — on the first confirmed cross-agent handoff need (on-demand, not pre-created) |
| **Update authority** | Project-dependent. P/M Lead directed. Chain of command: Jay → Sage → Sophia → P/M Lead → relevant agents |
| **Notification** | Project and task dependent. Chain of command ensures consuming agents are informed when a shared artifact changes |
| **Structure** | Project-defined — no prescribed subfolder pattern at baseline. Sage establishes structure based on the project's collaboration map |
