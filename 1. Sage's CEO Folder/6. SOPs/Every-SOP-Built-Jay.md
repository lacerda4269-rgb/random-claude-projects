## What This Is

This is your guide to the document build process — what happens, what each stage means, and where you come in.

Every operational document in the project is built using this process. The document type (agent SOP, standard SOP, pipeline, guide, Master Guide, runbook) determines the source and format. The build cycle itself is always the same.

---

## Why This Process Exists

Building any operational document is high-stakes work. The document becomes a permanent operating standard — it governs how the team works on every future project. Getting it wrong compounds.

The build cycle is designed to catch errors from two directions: inside the document's domain (what the executor or agent knows) and outside it (what the rest of the team sees). Both passes happen before anything is finalized. You review and approve before any document closes.

## TodoWrite — How Multi-Step Work Gets Tracked

Before any build that involves 2+ steps that each produce or change something real, Sage creates a task list first. Items get crossed off as each step completes.

**The test:** Does each step leave something behind — a file that changed, a record that was created, a version that was updated? Yes → list first, then execute. No → no list needed.

Session summary updates and compacting actions are excluded from this rule.

## The 9 Stages — What Each One Means

| Stage | Name | What's Happening | Your Role |
|---|---|---|---|
| 0 | Paired Version Decision | Sage confirms whether this document type gets a plain-language version paired with it | Answer yes or no — it's documented either way |
| 0a | Source Scan | Sage searches all existing documents for references to this new document | None — Sage runs this |
| 0b | Identify the Source | Sage identifies what document type this is, then reads the foundation document before drafting. For agent SOPs specifically, Sage also checks whether the agent has direct workflow dependencies on other specialists — if the soul is missing the routing protocol for that cross-lane work, the build holds until the soul is corrected first | None — if a soul hold fires, Sage will surface it before drafting begins |
| 0c | Compile Input List | Sage gathers all inputs: source scan findings, Sophia's build notes, any type-specific standing items | None — preparation step |
| 1 | First Draft (Primary Perspective) | Inside/out pass — drafted from inside the document's domain, then reviewed by the primary reviewer; corrections made; V1 locked | None — reviewer-facing pass |
| 2 | Second Pass (External Perspective) | Outside/in pass — Sage reviews V1 from the outside; corrections made; V2 final; your version built from V2 (if Yes at Step 0) | None until your version is handed to you |
| 3 | Your Review | You review the final document and your plain-language version; corrections applied | **This is your gate** — you read, flag anything unclear or wrong, and approve |
| 4 | Backward Sweep | Sage checks all prior built documents to ensure nothing is out of sync | None — Sage runs this |
| 5 | Team Lessons Check | Sage asks all agents involved whether there are lessons to log | None — Sage and agents handle this |

## Your One Gate — Stage 3

Your formal role in this process is Stage 3. That is where you review both versions (if applicable) and approve before the build closes.

**What to look for when you review:**
- Does your plain-language version make sense without needing to read the full document?
- Does it cover what you need to know — what this document is, when it applies, what your role is?
- Anything missing, wrong, or confusing? Flag it — corrections happen here before the document is finalized.

The build does not close without your review at Stage 3. This is not optional.

---

## The Paired Version Decision (Stage 0)

At the start of every build, Sage will confirm whether you want your own version.

- **Yes** — your version is built at Stage 2 (from the final draft) and you review it at Stage 3. Hard Rule 13 applies from then on: whenever the document updates, your version updates in the same session.
- **No** — the build proceeds without a paired version. That decision is documented — it's a deliberate choice, not a gap. A paired version can always be added later if you change your mind.

Sage's judgment check applies both ways. If you say **Yes** but Sage thinks a paired version is unnecessary — the document is too simple to warrant one, or it already IS a plain-language document — she'll flag it with her reasoning before building one. If you say **No** but Sage thinks you'll need one — the document is complex, or it governs a process you interact with or approve directly — she'll make the case and try to change your mind. The call is yours; the pushback is part of the gate.

For some document types (like Simple Guides — the "your version" documents themselves), a paired version doesn't apply because the document already IS the paired version.

---

## Hard Rule 13 — What It Means for Every Document

Once your version of a document exists, both versions update together in the same session — every time, no exceptions. A change to either one triggers a sync of the other before the session closes.

This document pair (Every-SOP-Built.md + this document) is subject to Hard Rule 13 from creation.
