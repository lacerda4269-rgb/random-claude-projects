## 1. Purpose

The Master Library is the single source of truth for every tool, skill, resource, and built item evaluated for the operator's workflows. It is a folder system — not a single file — actively maintained, reflecting the current state of what has been evaluated: accepted, rejected, pending, or under development. No item enters the library without completing the full pipeline. No evaluation is suppressed, reversed silently, or left unrecorded.

The library exists to serve the operator's specific workflows. General merit is not a filing criterion. Every judgment is made against declared domains: Trading, AI/Coding, Automation, Content, Finance. Items that don't yet fit declared domains are not automatically discarded — they go to Section 9 (Jay's Shelf) for future consideration.

## 2. Roles and Ownership

**Sage — Orchestrator**
Owns the pipeline as a whole. Manages sequencing and handoffs across both tracks. Every submission enters the pipeline through Sage — she routes each item to the correct track and briefs the assigned agent. Escalates HOLD verdicts to the operator when needed. Approves overrides. Ensures the session does not close with pending evaluations or unfiled items. Does not fetch from the internet. Does not file items directly. Tracks all queued items that belong to specific agents until those agents are activated.

**Direct submission to an agent (bypassing Sage):** If the operator submits directly to Rose, Cosmo, or any other agent, the receiving agent routes the submission through Sage before proceeding. The pipeline does not start until Sage has confirmed the track assignment. This is not a gate on the operator — it is a structural requirement that keeps the pipeline record clean from the moment an item enters.

**Rose — Researcher**
Primary internet gateway. No other agent — including Sage — fetches external resources without routing through the research pipeline. All external research on operational resources originates with Rose; academic and deep-source investigation routes to Feynman. Evaluates items for relevance and quality before anything reaches Soren. Writes the report file for every item researched. Does not make security judgments. Handles all submissions on the Resource Track.

**Cosmo — Builder**
Receives build requests from any agent or the operator. Builds skills, MCPs, CLI tools, and any other items that require construction rather than sourcing. Maintains temp logging during every build — this log feeds the AAR. Does not make security judgments. Built items pass to Soren before any library entry.

**Soren — Security Manager**
Screens every item before it reaches Lexi — on both tracks. The security step is non-bypassable, regardless of who submitted the item or how it was sourced. Soren does not evaluate relevance or quality. Soren evaluates safety and integrity only. Issues one of three verdicts: CLEARED, HOLD, or CLEARED WITH NOTE. Items that HOLD do not move forward until the operator decides.

**Lexi — Librarian**
Receives cleared items via Sage after Soren's clearance. Soren does not interact with Lexi directly — all routing goes through Sage. Files report files into the correct category subfolder. Writes the Full Database entry and links it to the report file. Updates all three catalog files (Full Database + both ML Index Cards) in a single pass — never a partial update. Maintains all filing rules. Updates the library in the same session as any evaluation or change. The library does not carry forward pending edits. Library sections are not locked in — they can grow, combine, split, or be removed as the system evolves. When in doubt about where something belongs, ask Sage rather than guess.

**All Agents — Submitters**
Any active agent working on a project may submit a request to the library pipeline. This applies to the full roster: Nick, Todd, Pete, Rose, Soren, Lexi, Cosmo, Sophia, and Sage. Agents working in their lanes will encounter things the operator may not know about — resources, tools, or build needs specific to their domain. Those submissions are welcome and expected.

### Temp Agents and the Library

When a temporary agent is created for a project, a review is conducted post-AAR to determine whether that agent is kept. If kept, a second review takes place to determine whether the agent has a role in the Master Library pipeline. This is a standing rule — no permanent agent joins the roster without the pipeline question being asked.

## 3. Library Structure

### 3A. ML Folder Architecture

The Master Library is a folder system. The ML root contains one numbered admin folder (`0. ML Operations`) and one category subfolder per section. The admin folder is flat — it holds the two SOPs (Lexi's version + Jay's version), the blank research-report request form, and the three catalog files (Full Database + both ML Index Cards).

**Catalog Files (inside `0. ML Operations`):**

- **Full Database** (`3. Master Library Index — Full Database.md`) — the live shopping directory; the navigation layer for the entire library; find any item here first; ~63 items and growing
- **Quick Reference Index Card** (`4. ML Index Card — Quick Reference.md`) — fast-scan table across all 9 sections; columns: Item | Tier | Soren Status | Status; built and maintained by Lexi
- **C&P Summary Index Card** (`5. ML Index Card — C&P Summary.md`) — full C&P text per item in the same 9-section structure; one plain-English paragraph per item for quick operator review; built and maintained by Lexi

**Note:** The blank research-report request form (`2. Master Library - Research Report Request (Blank).md`) also lives in this folder — see Section 4. Submissions are made via Sage, directly in conversation or using the form; there is no separate standing submission-list file.

**Category Subfolders:**
One subfolder per section lives in the ML root (e.g., `1. Soul Folder/`, `2. Skills/`, `3. Command Line Interface (CLI)/`, etc.). For most sections, each category subfolder contains two subfolders:

- **Research Reports/** — Rose's report files are filed here permanently; reports never move or get deleted; this is the permanent research record for every item regardless of deployment status
- **Built/** — the deployed artifact (built skill, configured tool, installed MCP, etc.) is placed here when an item goes live; the report file stays in `Research Reports/` and is never moved

**Note — Section 1 (Soul Folder) has additional internal structure:** The `1. Soul Folder/` folder contains more than just `Built/` and `Research Reports/`. It also contains `0. Generic Soul Shells/`, `0a. Basic Built Agents/`, and flat per-agent folders named for the agent (e.g. `Soren the Security Boss/`, `Lexi the Librarian/`, ... `Pete and Python/`). These are storage folders for the built agent files themselves. `Research Reports/` and `Built/` remain the correct locations for report files and built artifacts in Section 1, as with all other sections.

**Report Files:**
Sophia (COO) writes a markdown report file for every item Rose researches (`report-[item-name].md`). Rose provides the research content and evaluation; Sophia writes and files the report document. These files are the full backing record for each item: what it is, what it does, relevance assessment, Soren's security pre-check, and recommendation. Lexi files each report into the relevant category's `Research Reports/` subfolder. Reports are permanent — they are never moved or deleted. When an item is deployed, the built artifact goes into the `Built/` subfolder; the report stays in `Research Reports/`.

**Report file internal structure:** Every report file follows this 9-section format:
1. Header block: URL, Date Researched, Researched by / Security pre-check / Report by, Already in Master Library?
2. What It Is
3. What It Does
4. Relevance to This Project
5. Master Library Assessment (proposed section, proposed tier, rationale)
6. Soren Security Pre-check (verdict, what was checked, notes)
7. Recommendation
8. **C&P Summary** — a single plain-English paragraph explaining what the item is and does, written for a non-technical reader. No jargon. No assumed knowledge. If a non-technical operator can read it and immediately understand what the thing does and why it matters, the C&P is complete. This section is mandatory — a report without a C&P section is an incomplete report. Rose provides this paragraph as part of her research delivery; Sophia includes it in the filed report.
9. Footer line (report by, session, date)

### 3B. Section List

The Master Library contains nine sections. Items are filed only in the section that matches what they actually are. Sections are not locked in — they can grow, split, combine, or be retired as the system evolves.

1. **Agents & Autonomous Systems** — full agent builds, autonomous runners, and multi-agent orchestration systems
2. **Skills** — reusable capability instructions for agents; invoke by name, no reinstall needed
2a. **Hooks** — automatic guardrails built and maintained by Cosmo; subordinate to Section 2 per AR 25-50 convention; `Built/` and `Research Reports/` subfolders
3. **CLI (Command Line Interface)** — command-line tools providing terminal-native access to platforms, services, and agents
4. **MCP Servers** — tools that connect Claude to external systems (file systems, APIs, platforms, databases)
5. **Plugins** — Claude-native plugins and extensions; add-ons that extend Claude's built-in capabilities
6. **Workflow Patterns** — reusable workflow designs, orchestration patterns, and process templates
7. **Reference & Ecosystem** — stable strategic foundations, architecture references, ecosystem maps; changes rarely
8. **Alternative Tools & Notable Ecosystem** — secondary-layer tools; notable ecosystem items worth knowing that are not core to the build
9. **Jay's Shelf** — items outside declared domains that don't fit anywhere yet; potential future use; items stay here until assigned to a section, promoted, or formally dropped; not a rejection — a holding area

## 4. Submission Intake

Submissions may come from the operator or from any active agent. Any agent working on a project may identify a resource, tool, or build need within their domain and submit it through the pipeline.

**What can be submitted:**
- URLs (external tools, MCP Servers, frameworks, platforms)
- Skill requests (build a new skill for a specific need)
- MCP Server requests (build or configure a new integration server)
- CLI requests (build or configure a new CLI tool)
- New agent requests (temporary or standing team addition)
- Repeating research templates (when Rose runs the same search pattern repeatedly, the template itself becomes a library candidate)
- Any other resource or built item that may serve the operator's workflows

**Two tracks:** Submissions route to one of two tracks based on what is being submitted.

- **Resource Track** — an existing external item (URL, tool, framework, platform): routes to Rose for research
- **Build Track** — something that needs to be built (skill, MCP Server, CLI tool, template): routes to Cosmo for construction

Both tracks converge at Soren before anything reaches Lexi.

**Submission form (optional):** A submission form (`2. Master Library - Research Report Request (Blank).md`, in `0. ML Operations`) using the 5 Ws format may be used to provide context for Rose or Cosmo. The more context provided, the faster and more targeted the research or build. The 5 Ws don't need to be complete — partial context is still useful. The form is a helper, not a gate.

**Operator context for Resource Track:** Operator or agent context is a starting point, not a verdict. Rose researches every item fully regardless of what context is provided.

**How to submit:** Drop URLs and build requests to Sage directly in conversation, or use the blank research-report request form (`2. Master Library - Research Report Request (Blank).md`) to provide 5 Ws context. Either way, Sage routes each item to the correct track and briefs the assigned agent. There is no separate standing submission-list file — Sage captures every submission regardless of how it arrives.

## 5. Pipeline — Full Sequence

### Track A: Resource Pipeline (URL / existing external tool)

#### Step 0: Intake — Routing

1. **Submission received** — from the operator or any active agent, directly in conversation or via the blank research-report request form. Multiple items may arrive together as a list.
2. **Sage confirms the track** — each submission is reviewed and confirmed as Resource Track (existing external item) or Build Track (something to be built). Mixed-track batches are split and routed to Rose and Cosmo separately.
3. **Sage briefs Rose** — Sage passes the submission to Rose with any context provided at intake. Research begins at Step 1.

**Batching note:** When a list of items is submitted together, Sage confirms all tracks before Rose begins research on any of them.

---

#### Step 1: Rose — Research, Evaluation, and Report

Rose receives the submission. For each item:

1. **Already in Master Library?** — Before beginning research, Rose checks whether the item already exists in the Master Library Index. If it does: the task is a verification — the report confirms or corrects the existing entry rather than creating a new one. If it is net-new: proceed through the full research steps below.
2. Fetch and read the primary source in full — the complete README, documentation page, or landing page. Search snippets are not an acceptable basis for evaluation. This is non-negotiable.
3. Run follow-up searches if the primary source is insufficient. Fill specific gaps: star count, license, maintainer identity, known limitations, behavior in practice.
4. Establish what the item actually is: category (MCP Server, agent skill, CLI tool, framework, platform, extension), precise function, creator/maintainer, maturity signals, maintenance status.
5. Apply the relevance filter — two layers:
   - **Layer 1 — Library Fit:** Does the item serve the operator's declared workflows? (Trading, AI/Coding, Automation, Content, Finance.) If it does not serve any declared domain, it does not automatically stop — it routes to Section 9 (Jay's Shelf). Rose flags it clearly: "Outside declared domains — recommend Jay's Shelf."
   - **Layer 2 — Trading Fit (only when applicable):** If the item is trading-related, evaluate against: strategy development/validation, backtesting, market data, broker/platform integration, trading content. Skip this layer entirely for non-trading items.
6. Apply five evaluation dimensions:
   - **Integration Fit** — does it work with the operator's existing stack?
   - **Phase Fit** — is it usable now (Phase 1), usable soon (Phase 2), or later (Phase 3)?
   - **Maturity Signal** — evidence of stability and active maintenance
   - **Overlap Check** — does it duplicate something already in the library? If yes: is it a strict upgrade, a complement, or a redundant alternative?
   - **Risk or limitation flags** — anything worth documenting before acceptance
7. Assign domain tags: one or more of Trading, Coding, Automation, Content, Finance. Trading is always tagged even when the item also serves other domains.
8. Issue a verdict on fit and relevance. Eight outcomes:
   - ADD — CORE LIBRARY
   - ADD — SKILL
   - ADD — REFERENCE
   - ADD — NOTABLE/ECOSYSTEM
   - SKIP — NOT RELEVANT
   - SKIP — SUPERSEDED
   - PENDING — OUTSIDE DECLARED DOMAINS (routes to Section 9)
   - REASSESS (item needs more information or a future check-in)
9. If accepted: assign Priority rank relative to other items in the same section, and Phase Fit ranking.
10. Sophia writes the report file: `report-[item-name].md` — containing all research, findings, evaluation dimensions, Rose's verdict, recommendation, and C&P Summary. Rose provides all research content including the C&P paragraph; Sophia writes and files the document.

**Batching rule:** When multiple items are submitted together, Rose researches each independently, completes all evaluations, then passes the full batch to Soren as a single handoff. Items are not passed one at a time.

**Operator override handling:** If the operator overrides a SKIP verdict — he wants to add something Rose recommended against — Rose respects the override, documents it, and flags it in the report file as "Override: Yes — [operator's reason]." The entry goes forward through the full pipeline. Rose has the authority to argue her position clearly before an override is finalized. If Rose presents solid reasoning, the operator will weigh it. The operator never makes these decisions in a vacuum — he respects every agent's insight, reasoning, recommendations, suggestions, and ideas. Agents simply need to present a valid reason. He can always change his mind.

Rose passes the completed report file (containing all research, Rose's verdict, domain tags, and any override flag) to Soren. The report file IS the evaluation package — there is no separate handoff document.

---

#### Step 2: Soren — Security Screening (Resource Track)

Soren receives Rose's report file. Security screening happens before any item reaches Lexi. This step cannot be skipped, shortened, or deferred.

Soren screens for:
- Malicious code or payloads embedded in linked resources
- Suspicious provenance: unknown developer, no verifiable source, anonymous repository
- License issues: licenses that create legal or operational risk for the operator
- Data exfiltration risk: tools that transmit data to unverified or unknown destinations
- Dependency chains: does this item pull in other packages that have not been cleared?
- Integrity mismatch: does the tool do what it claims, or does it do something else?

Soren issues one of three verdicts:

- **CLEARED** — item is safe to file; passes to Lexi with clearance note attached
- **HOLD** — a specific concern has been identified; item is not filed; Sage is notified immediately; operator decides whether to investigate further or drop
- **CLEARED WITH NOTE** — item is safe but carries a flag worth documenting. Item passes to Lexi with the note attached.

**Hard rule:** If Soren has any doubt, the verdict is HOLD. Partial clearances do not exist.

**Soren writes her verdict into the report file:** Soren writes the verdict, what was checked, and any notes into the "Soren Security Pre-check" section of the report file. The report file — with the verdict embedded — is then passed to Lexi. This is how the report becomes the complete backing record.

**CLEARED WITH NOTE:** The note goes into the Soren Security Pre-check section of the report file AND into the Full Database entry. Both places.

HOLD items go to Sage. Sage reports to the operator. The operator decides: investigate further, drop, or revisit later.

> **HOLD — Report File Handling:** When Soren issues a HOLD verdict, the report file remains in Soren's care until the operator resolves the decision. The item does not move to Lexi. The report file is not filed anywhere. Soren holds the file pending operator direction (investigate further, drop, or revisit later).

> **[OPEN — OPERATOR DECISION REQUIRED]**

Does the whole item HOLD pending clearance of its dependencies? Do the dependencies get submitted separately through the pipeline? Does Soren flag each dependency individually? This is currently undefined. The outcome path needs to be established before Soren first encounters an item with uncleared dependencies.

### Track B: Build Pipeline (skill / MCP Server / CLI tool / other item to be built)

#### Step 0: Intake — Routing

1. **Build request received** — from the operator or any active agent, directly in conversation or via the blank research-report request form.
2. **Sage confirms the track** — confirmed as Build Track. Sage reviews the request, clarifies any gaps if needed, and briefs Cosmo.
3. **Sage briefs Cosmo** — Sage passes the build request to Cosmo with any context provided. Build begins at Step 1.

---

#### Step 1: Cosmo — Build and Temp Logging

Cosmo receives the build request. For each item:

1. Review the request fully — understand what is being built, who submitted it, what workflow it serves, and any constraints.
2. Clarify with the submitter (or Sage) if the request is unclear before beginning. Guessing wastes the build.
3. Build the item.
4. Maintain temp logging throughout the build: what was built, what decisions were made, what was discarded, what remains unresolved. This log is the record that feeds the AAR discussion. It is not optional.
5. Complete the build and pass the item — with the temp log and a brief build summary — to Soren.

**AAR gate:** Anything Cosmo builds is temporary until the AAR confirms it is kept. The temp log is what the team reviews at the AAR to make the keep/drop decision. If kept, the item becomes a permanent library entry. If dropped, the temp log is archived.

Cosmo passes the completed build package (built item + temp log + build summary) to Soren.

---

#### Step 2: Soren — Security Screening (Build Track)

Soren receives Cosmo's build package. The security step applies to built items exactly as it applies to external resources. Built items may pull in dependencies, external packages, or external calls that require screening.

Same three verdicts apply: CLEARED, HOLD, CLEARED WITH NOTE.

**Hard rule:** If Soren has any doubt, the verdict is HOLD. Partial clearances do not exist. Built items are not exempt from this rule — Cosmo builds with good intent; Soren verifies.

**Soren writes her verdict into the report file:** Soren writes the verdict, what was checked, and any notes into the "Soren Security Pre-check" section of the report file. The report file — with the verdict embedded — is then passed to Lexi.

**CLEARED WITH NOTE:** The note goes into the Soren Security Pre-check section of the report file AND into the Full Database entry. Both places.

HOLD items go to Sage. Sage reports to the operator.

### Filing Rules (Both Tracks)

Lexi receives cleared items routed by Sage — either CLEARED or CLEARED WITH NOTE, with the report file (Soren's verdict embedded). Soren does not hand off to Lexi directly. The routing always goes through Sage.

Lexi files each item according to eight rules:

1. **Replace, don't accumulate** — when a better version of something exists, replace the old entry; do not keep both
2. **Section integrity** — items go in the section that matches what they actually are; nothing is forced into a section for convenience; when unsure, ask Sage
3. **Honest gaps** — if no solution exists for a known need, document the gap explicitly with the current workaround and what to watch for
4. **Phase discipline** — installation and adoption order reflects dependency and readiness, not importance; phase assignments do not change unless the dependency structure changes
5. **Sync on every session** — the Full Database is updated in the same session as any evaluations or changes; the session does not close with pending edits
6. **Corrections are public** — when an evaluation was wrong, the correction is stated explicitly in conversation and reflected in the record; errors are never silently fixed
7. **Verdicts are specific** — every accepted and rejected item has a reason that references the operator's actual workflows; "not relevant" alone is not sufficient
8. **Overrides are explicit** — when the operator overrides a skip verdict, the entry clearly states "Override: Yes" with the reason in plain language

**Filing sequence for a new item:**
1. Lexi files the report file (`report-[item-name].md`) into the category's `Research Reports/` subfolder; it stays there permanently
2. Lexi writes the Full Database entry for the item; the Report column links to the report file
3. Lexi updates both ML Index Cards in the same pass — Quick Reference and C&P Summary. All three catalog files (Full Database + both Index Cards) updated as a single operation. No separate step.

**Cross-reference trigger:** The three-file catalog update (steps 2–3 above) fires when an item is complete and no further changes are expected in that session — after Soren clears and Lexi files. It does not fire mid-session on items still in motion. For souls: trigger fires when the soul is built and accepted. For reports: trigger fires when the report is filed.

**Ranking on insert:** Accepted items are inserted in ranked order — by priority relative to other items in the same section, and by Phase Fit.

**Date on insert:** When Lexi files any item, the date it was added to the library is recorded in the entry. This date is not updated on subsequent edits — it reflects when the item first entered the library.

**Rejected (Unsafe) items:** Items with HOLD verdicts that the operator decides to drop are recorded in the Full Database in the relevant section as a rejected entry — with the reason stated plainly. They are never deleted from the Full Database. This is the permanent record that the item was seen and rejected.

**Jay's Shelf items:** Items outside declared domains, or items the operator submits that don't fit current sections, go to Section 9 (Jay's Shelf). They are not rejected — they are held for future consideration. Lexi reviews them at each periodic review for potential promotion to a main section.

**CLEARED WITH NOTE items:** The note from Soren is included in both the Soren Security Pre-check section of the report file and the Full Database entry.

**Build Complexity — mandatory field for all new entries (Item 136 — Session 192):** Every new Full Database entry filed from Session 192 forward must have the Build Complexity column populated with a plain-language estimate before the entry is considered complete. A blank or `—` value without `(retrofit pending)` is an incomplete entry — do not mark it filed until this field is filled.

**RETROFIT ruling (Session 175 — Jay direction):** Existing entries already in the Full Database with `— (retrofit pending)` in the Build Complexity column are NOT required to be backfilled. They pick up the mandatory requirement at their next natural update — the next time the entry is reviewed, corrected, or otherwise touched for any reason. This exception applies only to entries cataloged before Session 192. All entries from Session 192 forward require Build Complexity populated at time of filing, no exceptions.

## 6. Periodic Review

The library is maintained as a living document. Two review types are in effect:

**Scheduled review — minimum every 4 weeks:**
A full A-Z review of all library entries. Rose re-researches any entry that may have changed: new versions, forks, replacements, deprecations, security developments, META shifts. Lexi updates any entries that need correction. Sage coordinates and reports the outcome to the operator.

The date the last full A-Z check was completed is recorded in the Full Database header. Lexi updates this date at the close of every scheduled review. During the review, Lexi also checks the date-added field on each entry — items that have been in the library longer without a review pass get priority attention from Rose.

**Trigger-based review — as needed:**
Triggers:
- A major update to a tool already in the library
- A potential replacement tool surfaces
- A build phase is completed
- An entry feels outdated relative to current state
- A new workflow domain is added
- Several sessions have passed without a review of a particular section

When a trigger occurs, Rose re-researches the affected item, compares it against the current library entry, and issues a recommendation: confirm (entry stands), update (entry is amended), or replace (entry is swapped for a better option). Lexi executes the recommendation. The change is reflected in the same session.

## 7. Operating Principles

These are non-negotiable commitments that govern every evaluation and every filing decision.

1. **Read before judging** — no item is evaluated without reading the actual source. No exceptions.
2. **Reason drives the verdict, not metrics** — star count and popularity are signals, not verdicts. The judgment is about fit with the operator's specific workflows.
3. **Specificity over generality** — every judgment is made against the operator's declared workflows, not general merit or industry reputation.
4. **Sequence awareness** — every recommendation accounts for what is already in place and what must come first. Phase Fit is always considered.
5. **Transparency on errors** — mistakes are corrected openly. The record is updated to reflect corrections. Silent fixes are not permitted.
6. **Lean over comprehensive** — 30 well-chosen, well-reasoned entries is a better library than 100 entries where half are redundant. Quality and fit over volume.
7. **Future domains are real** — the operator's work will grow into areas not yet declared. Items that don't fit today may fit tomorrow. Jay's Shelf is not a consolation prize — it is a real holding section for real potential.

## 8. Edge Cases and Standing Rules

**TodoWrite — execution sequences:** Before any multi-step ML operation with 2+ steps that produces or changes something persistent (pipeline runs, catalog updates, batch filing, report sequences, agent brief + execution + handoff chains), a TodoWrite task list is the first action. Items crossed off as each step is completed. Trigger test: does each step produce or change something that persists after the session? Yes → TodoWrite required. No → not required. Excludes: session summary updates and compacting actions.

**No internet access outside Rose:** Sage, Lexi, Soren, Cosmo, and all other agents do not fetch from the internet. Any need for external research is a task for Rose.

**No filing without clearance:** Lexi does not file any item that has not passed through Soren. There is no shortcut for items that "seem obviously fine." This applies to both tracks.

**HOLD escalation:** Every HOLD is reported to Sage. Sage reports to the operator. The item sits until the operator decides. It does not expire silently.

**Session close discipline:** Sage confirms before closing any session that no evaluations are pending, no items are unfiled, and the Full Database reflects the current state of all work done in that session.

**Override documentation:** Operator overrides are always recorded in the Full Database entry. The override reason is written in plain language. Future agents reading the entry will know it was a deliberate operator decision, not an evaluation outcome.

**Reassess items:** Items issued a REASSESS verdict are tracked by Sage in a holding state. They are not filed in the main sections and not added to any rejection record. Sage tracks them for future review when the trigger condition is met.

**Jay's Shelf items:** Items routed to Section 9 are reviewed at every scheduled periodic review. Lexi checks whether any Jay's Shelf item can be promoted to a main section based on new information, new project direction, or operator guidance.

**Temp agent pipeline review:** When a temporary project agent is kept after AAR, a review is conducted to determine whether the agent has a role in this pipeline. The question is always asked — never assumed.

**Built item ownership:** Everything Cosmo builds is tracked in the temp log until the AAR keep/drop decision is made. If kept, it becomes a permanent library entry; the report file is filed in the category's `Research Reports/` subfolder and the built artifact in the `Built/` subfolder. If dropped, the log is archived. Nothing built and kept is left unrecorded.
