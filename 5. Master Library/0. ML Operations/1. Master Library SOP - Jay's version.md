## What Is the Master Library?

The Master Library is a folder system — not just a single file — that tracks every tool, skill, resource, and built item we've looked at or created for your workflows. Think of it as a permanent, organized record: what we've added and why, what we passed on and why, what we're watching for the future, and what's waiting to find its place.

It exists so that nothing gets lost, nothing gets added carelessly, and you can always look at it and understand exactly what's in your system and how we got there.

---

## How the Folder System Is Organized

The Master Library is a folder in your project. Inside it, there's one numbered admin folder (`0. ML Operations`) that holds the SOPs, the blank request form, and the catalog files, and then there's a subfolder for each category of item.

**The catalog files** (inside `0. ML Operations`):

**The Full Database** is the live shopping directory. This is where you go first when you want to find anything in the library. It's organized by category, and every item has a status, a tier, a Soren verdict, a source link, and a link to the full write-up. Think of it like a mall directory — it tells you where everything lives without making you search through every store.

**The Quick Reference Index Card** is a fast-scan table across all 9 library sections. Columns: Item | Tier | Soren Status | Status. Use it when you want a quick scan of what's in the library without opening the Full Database. Built and maintained by Lexi.

**The C&P Summary Index Card** is the full C&P text for every item in the same 9-section structure — one plain-English paragraph per item. Use it when you want to quickly understand what something does without opening the individual report files. Built and maintained by Lexi.

**The category subfolders:**

For each section of the library — Agents, Skills, CLI tools, MCP Servers, and so on — there's a corresponding folder. Each folder has two subfolders inside it: **Research Reports** and **Built**.

When Rose researches something and we file it, the report goes into **Research Reports** — permanently. It never moves. That folder is the permanent research record for every item, regardless of what happens next. Once you actually install or use something, the deployed artifact goes into **Built**. The report stays in **Research Reports**.

**A note on Section 1 (Soul Folder):** The Soul Folder has a bit more internal structure than the other sections — it contains additional folders for the built agent files themselves (`0. Generic Soul Shells/`, `0a. Basic Built Agents/`, and flat per-agent folders named for the agent, e.g. `Soren the Security Boss/` through `Pete and Python/`). Report files for Section 1 still go into `Research Reports/` and built artifacts into `Built/`, the same as every other section; the extra folders are just where the actual agent files live.

## The Report Files

Every time Rose researches something, a report file gets written for it — a markdown document named something like `report-github-cli.md`. Rose provides all the research content and evaluation; Sophia (COO) writes the actual document and files it. This is a collaborative step — Rose does the digging, Sophia writes it up.

The report file is the full write-up: what the tool is, what it does, how it fits your workflows, what Soren flagged or cleared, and what the recommendation was.

Every report file follows the same structure:
1. Header block (URL, date researched, who researched / who did the security check / who wrote the report, whether it was already in the Master Library)
2. What It Is
3. What It Does
4. Relevance to This Project
5. Master Library Assessment (proposed section, proposed tier, rationale)
6. Soren Security Pre-check (verdict, what was checked, notes)
7. Recommendation
8. **C&P Summary** — a single plain-English paragraph: what is this thing and why does it matter? No jargon. Written so anyone can understand it without any background. This section is mandatory — Rose writes it as part of her research, Sophia includes it in every report.
9. Footer line

The Full Database links to each item's report file. So if you're browsing the Index and want to go deeper on something, you can click straight through to the full picture. This is also the thing that makes the library trustworthy over time — the decision trail is always there.

## Who Runs It?

Five agents own different pieces of this process.

**Rose** is your researcher. She's the only one who goes to the internet. When anyone submits a tool or resource for consideration, Rose looks it up properly — reads the full documentation, checks who built it, figures out what it actually does, and decides whether it belongs in your system. Before she starts researching, she also checks whether the item is already in the Master Library — if it is, the job is to verify or update the existing entry, not create a new one. She has real authority here: if she thinks something is a bad fit, she'll tell you and explain why. You're always the final decision-maker — but she'll make sure you have the full picture first.

**Cosmo** is your builder. When someone requests something that needs to be built — a new skill, an integration tool, a CLI, a template — that goes to Cosmo. He builds it, keeps a log of everything he did, and passes the finished item to Soren before it goes anywhere near the library.

**Soren** is your security manager. After Rose evaluates something or Cosmo builds something, Soren checks whether it's safe. She's looking for things like hidden risks, sketchy sources, license problems, or tools that might send your data somewhere you don't want it going. Nothing gets added to the library without her sign-off. No exceptions — not even for things that seem obviously fine. When she issues a verdict, she writes it directly into the report file's security section, so the record is complete and there's no separate document to chase.

**Lexi** is your Master Librarian. She takes everything that's been researched or built, cleared by Soren, and routed to her by Sage, and files it in the right category subfolder. When she files an item, she updates all three index files in one pass — the Full Database, the Quick Reference Index Card, and the C&P Summary Index Card. She doesn't update the catalog while an item is still being evaluated; she waits until it's fully processed and handed to her. The sections she manages aren't locked in forever — they can grow, split, or change as your work evolves.

**Sage** runs the whole process. She makes sure handoffs happen, sessions close clean, and any issues get flagged to you when a decision is needed.

**Sophia** is your Chief Operating Officer (COO). She writes and files the report documents — Rose provides the research, Sophia puts it into the report file format and hands it off to Soren.

## How Does Something Get Into the Library?

There are two paths depending on what's being submitted.

---

### Path 1: Resource Path (a URL or existing external tool)

**Step 1 — Someone Submits It**

You or any agent gives a URL to Sage — directly in conversation, or using the optional request form (`2. Master Library - Research Report Request (Blank).md`, in `0. ML Operations`) that walks through the 5 Ws (Who, What, Where, When, Why) to give Rose a head start. If you want to add context — what you think it does, where you found it, why you're interested, a concern — that's helpful but never required. Rose will research it fully regardless.

You can submit anything: tools, platforms, MCP Servers, CLI tools, frameworks, research templates, anything you've come across. If it doesn't fit your current declared areas (Trading, AI/Coding, Automation, Content, Finance), it doesn't get discarded — it goes to Jay's Shelf. More on that below.

**What happens next:** Sage routes the submission to the correct track (Resource Track for existing tools and URLs, Build Track for something to be created) and briefs Rose or Cosmo.

**If you go directly to Rose or another agent:** That's fine — they'll loop Sage in automatically and Lexi will create the pipeline record before any work starts. Nothing is lost and no one is skipped — it just happens in the right order.

**Step 2 — Rose Researches It and Sophia Writes the Report**

The first thing Rose does is check whether the item is already in the Master Library. If it is, the work is a verification — confirm or correct the existing entry. If it's new, she reads the actual documentation. Not a summary, not a headline — the real source. She figures out what it is, who built it, whether it's actively maintained, and whether it fits your workflows. Sophia then writes the full report file that becomes the permanent backing record for this item.

Rose will give you one of these verdicts:
- Add it (to a specific section)
- Skip it (not relevant or already superseded)
- Put it on Jay's Shelf (outside your declared areas for now, but potentially useful later)
- Reassess it later (needs more information or isn't ready yet)

**Step 3 — Soren Clears It**

Before anything gets filed, Soren checks it for safety. She can come back with one of three answers:

- **Cleared** — safe to add; she writes her verdict into the report file and passes it along to Lexi
- **Hold** — something looks off; it doesn't move forward; Sage tells you and you decide what to do
- **Cleared with a note** — safe to add, but there's something worth knowing (like "this is maintained but not updated frequently — keep an eye on it"); Soren writes the note into the report file's security section, and Lexi also includes it in the Full Database entry

If Soren has any doubt at all, it's a Hold. There's no partial clearance.

**Step 4 — Lexi Files It**

Sage routes the cleared report to Lexi. Lexi puts it in the right category's Research Reports folder permanently and updates all three index files in one pass — the Full Database, the Quick Reference Index Card, and the C&P Summary Index Card. When she files an item, she records the date it was added — so the library always shows when each entry first came in. When an item is deployed, the deployed artifact goes into the Built folder; the report never moves from Research Reports.

**Step 5 — Lexi Closes the Batch**

Once every item in the submission batch has reached a final status (Filed, Rejected, Jay's Shelf, Skipped), the batch is closed — Sage confirms all items are resolved and records batch completion in the session record. This is what closes out the full RRR pipeline for that batch. If any item is still on Hold when the rest finish, the batch stays open until you make a call on it.

---

### Path 2: Build Path (something that needs to be created)

**Step 1 — Someone Submits a Build Request**

You or any agent submits a request for something to be built — a new skill, a tool, an MCP Server, a CLI, a research template. You describe what you need and what problem it solves. The more context the better, but Cosmo will ask if anything's unclear. Sage routes the request to Cosmo.

**Step 2 — Cosmo Builds It**

Cosmo takes the request and builds it. While he's building, he keeps a running log of every decision he made — what he built, what he tried and set aside, what's still unresolved. That log matters: it's what the team reviews at the AAR to decide whether the built item gets kept permanently.

Everything Cosmo builds is temporary until the AAR confirms it's a keeper.

**Step 3 — Soren Clears It**

Even things Cosmo built go through Soren. A build can pull in outside dependencies or make external calls — Soren checks all of that. Same rule applies: any doubt means a Hold, no partial clearances. She writes her verdict into the report file's security section, and the report — with the verdict embedded — is what Lexi receives.

**Step 4 — Lexi Files It**

Same as the Resource Path — Sage routes the cleared item to Lexi. Lexi files it in the right folder, updates all three index files in one pass, and records the date it was added.

## What's Actually In the Library?

The library has nine sections. These are not fixed forever — they can grow, split, or change as your work grows.

1. **Agents & Autonomous Systems** — full agent builds, autonomous runners, and multi-agent orchestration systems.
2. **Skills** — files that give your agents specialized knowledge. These get installed directly into your skills directory and can be invoked by name without reinstalling.
2a. **Hooks** — automatic guardrails built and maintained by Cosmo; subordinate to Section 2 per AR 25-50 convention. They enforce rules automatically — you don't invoke them, they fire on their own. Each hook folder has the same `Built/` and `Research Reports/` subfolders as every other section.
3. **CLI (Command Line Interface)** — command-line tools that give your agents terminal-native access to platforms, services, and other agents. CLIs are the preferred build choice before reaching for an MCP Server.
4. **MCP Servers** — the tools that connect your agents to external systems (trading platforms, databases, web services, file systems).
5. **Plugins** — Claude-native extensions and add-ons that extend Claude's built-in capabilities.
6. **Workflow Patterns** — reusable workflow designs, orchestration patterns, and process templates. The how-to recipes.
7. **Reference & Ecosystem** — stable reference material; foundational stuff that doesn't change often.
8. **Alternative Tools & Notable Ecosystem** — a secondary layer: notable tools worth keeping an eye on even if they're not core to the build.
9. **Jay's Shelf** — things that don't fit your current declared areas but might be useful for something you haven't thought of yet. This is not a rejection. It's a holding area. As your work grows into new directions, things in here may get promoted to a main section. And yes — this section exists specifically because your mind works the way it does. The WTF submissions have a home now.

## What Happens When Something Needs a Second Look?

The library isn't set-and-forget. We run two types of reviews:

**Scheduled review — at least every 4 weeks:**
Rose goes through the full library, A to Z. She checks for updates, new versions, better alternatives, deprecated tools, and any META shifts that might affect how things are categorized. Lexi updates anything that needs it. Sage reports back to you.

The Full Database header tracks when the last full A-Z check was completed — so you always know how fresh the review is. And every item in the library shows the date it was added, which helps Rose and Lexi prioritize entries that have been sitting longest without a proper look.

**Triggered review — whenever something comes up:**
If a tool gets a major update, a better option surfaces, a build phase completes, or an entry just feels stale — we don't wait for the 4-week mark. Rose re-researches it and Lexi updates it that same session.

You can also trigger a review anytime. If something feels off or you've found a potential upgrade, just say so.

## What Happens When You Override a Recommendation?

You're always the final decision-maker.

If Rose recommends skipping something and you want to add it anyway, that's your call. The entry will be added with a note that says it was an operator override and records your reason.

That said — Rose has real authority to argue her position. If she thinks something is a bad fit or a risk, she'll tell you clearly and make her case. You'll have the full reasoning before you decide. If she makes a solid argument, she expects you to weigh it honestly. The final call is still yours.

What the team should also know: you never make these calls in a vacuum. You respect everyone's insight, reasoning, recommendations, suggestions, and ideas — that's not just a courtesy, it's how you actually operate. An agent doesn't need a perfect argument to be heard; they just need to present a valid reason. And you can always change your mind. Present your case clearly and you'll get a fair hearing.

Your WTF submissions — the things outside your current declared areas that Rose would normally flag as "not relevant" — don't get rejected. They go to Jay's Shelf. That's not a consolation prize. It's recognition that your vision is bigger than your declared domains right now, and that's fine.

## The Principles Behind All of This

A few commitments that never change, no matter what we're evaluating:

- **We read before we judge.** No item gets evaluated based on a headline or a summary. Rose reads the actual source.
- **The reason matters more than the popularity.** A tool with a million stars that doesn't fit your workflow doesn't belong here. A less-known tool that fits perfectly does.
- **Everything is judged against your specific work** — not general industry reputation.
- **Mistakes are corrected openly.** If an evaluation was wrong, we say so and fix the record. Nothing gets quietly changed.
- **Lean is better than comprehensive.** A tight library of well-chosen tools is more useful than a bloated one full of things that barely apply.
- **Future domains are real.** Your work will grow. Things that don't fit today may fit tomorrow. Jay's Shelf exists because of that.

## Build Complexity — A Required Field

Every new item added to the Full Database from Session 192 forward must have a Build Complexity note filled in before Lexi marks the entry complete. This is a required column, not optional.

**What it is:** A plain-language estimate of how complex it would be to install, integrate, or build this item. A few words or a sentence is enough — something like "Single install command, no configuration needed" or "Requires API key setup, environment variables, and a custom configuration file." The goal is to give you and the team a quick, honest signal before committing to any build.

**Older entries:** Library entries added before this standard went into effect (Session 192) have `— (retrofit pending)` in that column. Those are not being backfilled — they'll pick up this requirement the next time they're touched for any other reason (review, correction, version update). Everything new gets filled in at filing time, no exceptions.

## How Multi-Step Library Work Gets Tracked

Before any multi-step ML operation — pipeline runs, batch filing, report sequences, catalog updates, agent handoff chains — a task list is created first. Each item gets crossed off as it's completed.

**The test:** Does each step leave something behind — a file that changed, a record that was created, an entry that was updated? Yes → list first. No → no list needed.

Session summary updates and compacting actions are excluded.

## The Short Version

You or any agent submits something — a URL or a build request — to Sage, directly in conversation or via the request form. Sage routes it to the right track and briefs the assigned agent. If it's a resource, Rose researches it (and checks whether it's already in the library first), then Sophia writes the report file. If it's a build, Cosmo builds it. Either way, Soren checks it for safety and writes her verdict directly into the report file before anything gets filed. Lexi files the report into the right category folder and writes the Full Database entry. If something needs your input — an override decision, a hold that needs resolution, a Jay's Shelf item you want to revisit — Sage brings it to you. The library stays current, honest, and built entirely around your workflows.

Three catalog files work together inside the `0. ML Operations` folder: the Full Database for finding things, and the Quick Reference Index Card and C&P Summary Index Card for fast navigation and plain-English item summaries. The blank request form lives there too, for queuing what you want researched next.

That's the whole system.
